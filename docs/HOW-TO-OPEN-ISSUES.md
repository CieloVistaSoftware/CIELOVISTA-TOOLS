# How to Open an Issue

Creating well-defined issues is critical for tracking bugs, feature requests, and other tasks. This guide outlines the standard process for opening a new issue in the `CieloVistaSoftware/cielovista-tools` repository.

## When to Open an Issue

Open an issue for any of the following:
- **Bug Reports:** Something is broken or not working as expected.
- **Feature Requests:** A new idea or an enhancement to an existing feature.
- **Technical Debt:** A proposal to refactor, clean up, or improve the codebase.
- **Documentation:** A request to add, update, or fix documentation.

## Steps to Create an Issue

1.  **Go to the Issues Page:**
    Navigate to the main issues page for the repository:
    [https://github.com/CieloVistaSoftware/CIELOVISTA-TOOLS/issues](https://github.com/CieloVistaSoftware/CIELOVISTA-TOOLS/issues)

2.  **Click "New Issue":**
    Click the green "New issue" button on the right side of the page.

3.  **Write a Clear Title:**
    The title should be a concise summary of the issue.
    - **Good:** `Bug: Running Tasks view fails to refresh after killing a process`
    - **Bad:** `Tasks broken`

4.  **Write a Detailed Body:**
    The body of the issue is the most important part. Provide as much detail as possible. Use the following structure as a guide.

    **For Bug Reports:**
    - **Summary:** A brief overview of the problem.
    - **Steps to Reproduce:** A clear, numbered list of steps to trigger the bug.
    - **Expected Behavior:** What you expected to happen.
    - **Actual Behavior:** What actually happened.
    - **Screenshots/GIFs:** Visual evidence is extremely helpful.
    - **Environment:** (e.g., VS Code version, OS, extension version)

    **For Feature Requests:**
    - **Summary:** A brief overview of the proposed feature.
    - **Problem:** What problem does this feature solve? Why is it needed?
    - **Proposed Solution:** A detailed description of how the feature should work.
    - **Mockups/Screenshots:** (Optional but helpful) Visual designs of the proposed UI.
    - **Acceptance Criteria:** A checklist of what must be true for the feature to be considered complete.

5.  **Add Labels:**
    Use labels to categorize the issue. This is crucial for filtering and organization. Key labels include:
    - **`bug`**, **`feature`**, **`documentation`**, **`tech-debt`**
    - **Component Area:** `area:webview-ui`, `area:mcp-server`, etc.
    - **Component:** `comp:running-tasks`, `comp:doc-catalog`, etc.
    - **Priority:** `p0-critical`, `p1-high`, `p2-medium`, `p3-low`

By following this process, you ensure that every issue is immediately understandable, actionable, and properly tracked within our project management workflow.
